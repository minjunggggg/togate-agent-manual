로그인 페이지

Togate 사내앱의 인증 진입점입니다. 이메일/비밀번호 로그인과 자동 로그인, 미등록 계정 안내 플로우를 다룹니다.

마지막 업데이트 2026-05-19

📋 개요

로그인 페이지는 src/app/login/page.tsx 에 위치하며, "use client" 클라이언트 컴포넌트로 구현됩니다. 폼 상태는 react-hook-form으로 관리하고, 계정 존재 여부 확인은 Server Action(actions.ts)으로 처리합니다.

💡

설계 원칙: 로그인 폼 전체가 클라이언트 컴포넌트입니다. 인터랙션(폼 상태, 자동 로그인 체크박스, 다이얼로그)이 많아 서버 컴포넌트로 분리할 이점이 없습니다. 계정 존재 확인(checkUserExists)만 Server Action으로 처리합니다.

주요 기능

  • → 이메일 + 비밀번호 로그인 (Credentials Provider)
  • → JWT 세션 기반 인증 (Auth.js v5)
  • → checkUserExists 선행 검사 → 로그인 권한 없는 계정 다이얼로그 안내
  • → 자동 로그인 (rememberMe) 옵션
  • → 로그인 후 callbackUrl 리다이렉트
  • → sessionStorage 폼 임시저장 (LOGIN_DRAFT) — 새로고침 시 복원
  • → 앱 종료 / 로그인 권한 없는 계정 다이얼로그 UI

📁 파일 구조

로그인 관련 파일은 src/app/login/에 위치하며, 인증 설정은 src/lib/에서 관리합니다.

🧩 컴포넌트

page.tsx — 클라이언트 컴포넌트

TSXsrc/app/login/page.tsx
"use client"

import { useState, useEffect } from "react"
import { useForm, Controller } from "react-hook-form"
import { signIn } from "next-auth/react"
import { useRouter } from "next/navigation"
import { checkUserExists } from "./actions"

type LoginForm = { email: string; password: string }

const LOGIN_DRAFT = "login_draft"  // sessionStorage 키

export default function LoginPage() {
  const router = useRouter()
  const [callbackUrl, setCallbackUrl] = useState("/")
  const [autoLogin, setAutoLogin] = useState(false)
  const [dialog, setDialog] = useState<"exit" | "unknownUser" | null>(null)

  const { control, handleSubmit, watch, setValue, setError,
    formState: { errors, isSubmitting } } = useForm<LoginForm>({
    defaultValues: { email: "", password: "" },
  })

  const [watchedEmail, watchedPassword] = watch(["email", "password"])

  useEffect(() => {
    const params = new URLSearchParams(window.location.search)
    setCallbackUrl(params.get("callbackUrl") ?? "/")
  }, [])

  useEffect(() => {               // sessionStorage 복원
    const saved = sessionStorage.getItem(LOGIN_DRAFT)
    if (saved) {
      const draft = JSON.parse(saved)
      setValue("email", draft.email ?? "")
      setValue("password", draft.password ?? "")
    }
  }, [])

  useEffect(() => {               // 입력값 임시저장
    if (watchedEmail || watchedPassword)
      sessionStorage.setItem(LOGIN_DRAFT, JSON.stringify({ email: watchedEmail, password: watchedPassword }))
  }, [watchedEmail, watchedPassword])

  async function onSubmit({ email, password }: LoginForm) {
    const userExists = await checkUserExists(email)
    if (!userExists) { setDialog("unknownUser"); return }

    const result = await signIn("credentials", {
      email, password, rememberMe: String(autoLogin), redirect: false,
    })

    if (result?.error) {
      setError("password", { message: "비밀번호를 다시 입력해 주세요." })
    } else {
      if (!autoLogin) {          // 자동 로그인 미사용: 브라우저 세션 한정 처리
        sessionStorage.setItem("session_active", "1")
        document.cookie = "no_rm_session=1; path=/; SameSite=Lax"
      }
      router.push(callbackUrl)
      router.refresh()
    }
  }
}

actions.ts — Server Action

TSsrc/app/login/actions.ts
"use server"

export async function checkUserExists(email: string): Promise<boolean> {
  // DB에서 이메일로 사용자 조회 후 존재 여부 반환
  const user = await prisma.user.findUnique({ where: { email } })
  return user !== null
}
💡

왜 Server Action인가: checkUserExists는 DB에 직접 접근해야 하므로 Server Action으로 구현합니다. 클라이언트에서 await checkUserExists(email)로 호출하면 Next.js가 내부적으로 POST 요청을 처리합니다.

lib/auth.ts — NextAuth 설정

TSsrc/lib/auth.ts
import NextAuth from "next-auth"
import Credentials from "next-auth/providers/credentials"
import { prisma } from "@/lib/db"
import bcrypt from "bcryptjs"
import { authConfig } from "./auth.config"

export const { handlers, auth, signIn, signOut } = NextAuth({
  ...authConfig,
  providers: [
    Credentials({
      name: "credentials",
      credentials: {
        email:      { label: "이메일",    type: "email"  },
        password:   { label: "비밀번호",  type: "password" },
        rememberMe: { label: "자동 로그인", type: "text"    },
      },
      async authorize(credentials) {
        if (!credentials?.email || !credentials?.password) return null

        const user = await prisma.user.findUnique({
          where: { email: credentials.email as string },
        })
        if (!user) return null

        const isValid = await bcrypt.compare(
          credentials.password as string,
          user.password
        )
        if (!isValid) return null

        const prevLoginAt = user.lastLoginAt
        await prisma.user.update({
          where: { id: user.id },
          data:  { lastLoginAt: new Date() },
        })

        return {
          id:              user.id,
          email:           user.email,
          name:            user.name,
          role:            user.role,
          rememberMe:      credentials.rememberMe === "true",
          passwordChanged: user.passwordChanged,
          lastLoginAt:     prevLoginAt?.toISOString() ?? null,
        }
      },
    }),
  ],
})
💡

auth.config.ts 분리: session · pages · callbacks(JWT, session)는 auth.config.ts에 분리합니다. DB 접근(Prisma)과 bcrypt가 필요한 Credentials Provider 로직은 Node.js 전용 API를 사용하므로 proxy.ts와 공유 불가 — auth.ts에 직접 정의합니다.

💡

authorize 반환값: JWT에 포함할 커스텀 필드(role, rememberMe, passwordChanged, lastLoginAt)를 함께 반환합니다. lastLoginAt은 로그인 성공 직전 시각(이전 접속일)을 전달하기 위해 업데이트 전 값을 저장합니다.

✅ 유효성 검사

클라이언트에서는 react-hook-form으로 폼 상태를 관리하고, 제출 버튼은 이메일·비밀번호가 모두 입력된 경우에만 활성화됩니다. 서버 측 Zod 검증은 auth.config.ts의 Credentials Provider authorize에서 수행합니다.

🔒 보안

💡

미등록 계정 vs 비밀번호 오류 분리: 이 앱은 사내 임직원 전용이므로, 미등록 계정(관리자 초대 필요)과 비밀번호 오류를 의도적으로 구분해 안내합니다. 일반 공개 서비스에서는 계정 열거(Account Enumeration) 방지를 위해 두 에러를 동일한 메시지로 통일해야 합니다.

  • ✓ 비밀번호는 bcrypt (cost factor 12) 로 해싱
  • ✓ HTTPS 환경에서만 쿠키 전송 (Secure 플래그)
  • ✓ CSRF 토큰 — NextAuth 내장 보호 사용
  • ✓ proxy.ts에서 인증 게이트 + 초기 비밀번호 강제 변경 처리
  • ✓ AUTH_SECRET 32자 이상 랜덤 문자열 사용
  • ✓ 자동 로그인 미사용 시 no_rm_session 쿠키로 세션 범위 제한

⚙️ 환경변수

인증에 관련된 주요 환경변수는 아래와 같습니다. 전체 목록은 프로젝트 개요 → 환경변수에서 확인하세요.

변수설명
AUTH_SECRET JWT 서명·암호화 키. 32자 이상 랜덤 문자열 필수
AUTH_URL 로컬 개발 시 기준 URL. Vercel 배포 시 자동 주입 — 직접 설정 시 Preview URL 충돌 주의
AUTH_TRUST_HOST Vercel·역방향 프록시 환경에서 true로 설정
DATABASE_URL PostgreSQL 연결 문자열. Prisma 7 + @prisma/adapter-pg가 사용
💡

Vercel 배포 팁: AUTH_URL은 설정하지 않고 AUTH_TRUST_HOST=true만 설정하는 것이 권장됩니다.
