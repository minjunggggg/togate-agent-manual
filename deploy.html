🌐 Vercel 배포

Togate-agent는 Vercel에 연결된 GitHub 레포지토리를 통해 자동 배포됩니다. 코드를 푸시하면 별도 작업 없이 배포가 트리거됩니다.

브랜치Vercel 환경URL용도

developPreviewtogate-agent-git-develop-xxx.vercel.appQA / 스테이징
feature/* , fix/*PreviewPR별 고유 URL 자동 생성PR 단위 리뷰

💡

빌드 로그 확인: Vercel 대시보드 → 해당 Deployment → Build Logs 탭에서 실시간 로그를 확인할 수 있습니다. 빌드 실패 원인은 여기서 가장 먼저 확인하세요.

💡

Vercel 프로젝트 접근: vercel.com → 투게이트컴즈 팀 → togate-agent 프로젝트. 접근 권한이 없으면 외부 서비스 가이드의 담당자에게 초대 요청하세요.

⚙️ Vercel 환경변수 설정

Vercel 대시보드에서 환경변수를 설정합니다. 로컬의 .env.local은 Vercel에 자동 반영되지 않습니다.

BASHVercel CLI로 환경변수 추가 (선택)
# Vercel CLI 설치
npm install -g vercel

# 로그인 후 프로젝트 연결
vercel login
vercel link

# 환경변수 추가 (Production)
vercel env add AUTH_SECRET production
vercel env add DATABASE_URL production
변수명설정 환경비고
AUTH_SECRETProduction + Preview로컬과 다른 값 사용 권장
AUTH_TRUST_HOSTProduction + Preview반드시 true
DATABASE_URLProduction + PreviewPreview는 별도 DB 권장
GEMINI_API_KEYProduction + Preview
DEEPGRAM_API_KEYProduction + Preview
DEEPGRAM_PROJECT_IDProduction + Preview
NEXT_PUBLIC_VAPID_PUBLIC_KEYProduction + Preview
VAPID_PRIVATE_KEYProduction + Preview
VAPID_MAILTOProduction + Preview
CRON_SECRETProduction만Preview에서 Cron 불필요
AUTH_URL설정하지 않음Vercel에서 자동 주입. 직접 설정 시 Preview URL 충돌
🚨

AUTH_URL은 Vercel에 절대 설정하지 마세요. Preview 배포마다 URL이 달라지는데 AUTH_URL이 고정되면 OAuth 콜백 URL이 깨집니다. AUTH_TRUST_HOST=true만으로 충분합니다.

🗄️ 운영 DB 마이그레이션

스키마 변경이 포함된 배포 시 반드시 DB 마이그레이션을 먼저 진행합니다.

🚨

순서를 반드시 지키세요. 앱 배포 전에 마이그레이션을 먼저 완료해야 합니다. 앱이 먼저 배포되면 새 코드가 구 스키마를 참조해 런타임 에러가 발생합니다.

BASH운영 DB 마이그레이션 (prisma migrate deploy)
# 운영 DATABASE_URL을 환경변수로 설정 후 실행
DATABASE_URL="postgresql://..." npx prisma migrate deploy

# 또는 .env.production 파일 사용
npx dotenv -e .env.production -- prisma migrate deploy
순서작업담당
1스키마 변경사항 PR 리뷰개발자
2prisma migrate deploy 운영 DB 적용개발자
3마이그레이션 결과 확인 (Prisma Studio 또는 psql)개발자
4main 브랜치 병합 → Vercel Production 배포개발자
5운영 서비스 동작 확인개발자 + 담당자

↩️ 롤백

웹 앱 롤백 (Vercel)

BASHVercel CLI 롤백
# 특정 deployment로 롤백
vercel rollback [deployment-url]

# 또는 Vercel 대시보드 → Deployments → 이전 배포 → "Promote to Production"
⚠️

DB 롤백 주의: DB 마이그레이션은 자동 롤백되지 않습니다. 컬럼 추가와 같이 하위 호환되는 변경은 앱만 롤백해도 무방하지만, 컬럼 삭제·타입 변경이 포함된 경우 DBA와 협의 후 수동 롤백이 필요합니다.

📱 iOS 빌드 (Capacitor)

Capacitor 8을 통해 Next.js 앱을 iOS 네이티브 앱으로 래핑합니다. App Store 배포를 위해 Mac + Xcode 환경이 필요합니다.

⚠️

사전 요구사항: macOS + Xcode 15 이상 + Apple Developer 계정 (유료, 연 $99) 이 필요합니다. Windows 환경에서는 iOS 빌드가 불가합니다.

최초 iOS 프로젝트 세팅

BASHCapacitor iOS 초기 세팅
# 1. Next.js 정적 빌드
npm run build

# 2. Capacitor에 웹 에셋 동기화
npx cap sync ios

# 3. Xcode로 iOS 프로젝트 열기
npx cap open ios

기능 수정 후 iOS 앱 업데이트

BASH웹 코드 변경 → iOS 반영
# 1. 웹 빌드
npm run build

# 2. iOS에 동기화 (네이티브 플러그인 설치도 포함)
npx cap sync ios

# 3. Xcode에서 빌드 및 시뮬레이터/실기기 실행
npx cap open ios
# → Xcode: Product → Run (⌘R)

App Store 배포 순서

순서작업위치
1npm run build && npx cap sync ios터미널
2Xcode에서 ios/App/App.xcworkspace 열기Xcode
3버전·빌드 번호 올리기 (Info.plist 또는 TARGETS → General)Xcode
4Signing & Capabilities — 팀·프로비저닝 프로파일 확인Xcode
5Product → Archive → Distribute App → App Store ConnectXcode
6App Store Connect에서 심사 제출appstoreconnect.apple.com
💡

인증서 & 프로비저닝 프로파일: Xcode의 자동 서명(Automatically manage signing)을 활성화하면 Apple Developer 계정으로 자동 관리됩니다. 계정 접근은 외부 서비스 가이드의 담당자에게 문의하세요.

⚠️

PWA와 iOS 앱의 차이: 웹 푸시 알림(web-push + Service Worker)은 iOS Safari PWA에서 동작하지만, Capacitor 래핑 앱에서는 별도 네이티브 푸시 설정이 필요할 수 있습니다. 기능 수정 시 반드시 실기기 테스트를 진행하세요.

☑️ 배포 체크리스트

웹(Vercel) 배포 전

  • ✓ Vercel Production 환경변수 모두 설정 확인
  • ✓ 스키마 변경이 있으면 prisma migrate deploy 먼저 실행
  • ✓ Preview URL에서 주요 기능 동작 확인 완료
  • ✓ npm run build 로컬 빌드 오류 없음
  • ✓ Cron Job 경로 및 CRON_SECRET 설정 확인

iOS 배포 전

  • ✓ npm run build && npx cap sync ios 완료
  • ✓ 버전 번호 및 빌드 번호 업데이트
  • ✓ 실기기(iPhone) 테스트 완료
  • ✓ 인증서·프로비저닝 프로파일 유효 기간 확인
  • ✓ App Store Connect 앱 정보·스크린샷 업데이트
